# 知识库治理设计方案

## 1. 文档定位

本文以《企业级知识助教系统落地方案》为上位基准，定义课程资料解析完成之后的知识库治理边界。

| 边界 | 起点 | 终点 |
|---|---|---|
| 资料文件服务 | 上传原始文件 | 文档版本解析完成，`status = parsed` |
| 知识库治理 | 选择已解析资料版本 | 构建、发布、回滚可问答知识快照 |

`parsed` 不等于 `published`。资料文件服务不得直接决定学员可见范围；知识库治理不得修改原始文件内容、存储元数据或文件生命周期状态，只通过知识库版本和资料快照表达构建与发布关系。

## 2. 领域模型

MVP 采用“一门课程一个知识库”：

```text
course 1 --- 1 knowledge_base
knowledge_base 1 --- n knowledge_base_version
knowledge_base_version n --- n course_document_version
```

课程模块是资料组织和检索过滤维度，不单独创建知识库。后续如需一门课程多个知识库，再扩展知识库用途和授权，不改变文档版本模型。

知识库是课程下可发布、可问答的知识集合，不是文件夹或对象存储 Bucket。`knowledge_base.current_version_id` 指向当前提供学员资料访问和 AI 问答服务的已发布版本。

知识库版本是一次不可变的知识快照，固定：

- 纳入的文档版本。
- Chunk 和切分策略版本。
- 向量索引与关键词索引版本。
- 发布后的问答与检索日志归属。

### 2.1 自动初始化

课程创建成功时，在同一数据库事务中自动创建知识库基本记录和课程创建人的 `owner` 授权，不自动创建知识库版本。

| 字段 | 初始值 |
|---|---|
| `id` | 数据库生成 |
| `tenant_id` | 课程租户 ID |
| `course_id` | 新建课程 ID |
| `name` | `{课程名称}知识库` |
| `description` | 空 |
| `current_version_id` | 空，表示尚未发布 |
| `status` | `active` |
| `del_flag` | `0` |
| `create_by` | 课程创建人 |
| `create_time` | 当前时间 |
| `update_by` | 空 |
| `update_time` | 空 |
| `remark` | 空 |

初始化规则：

- `tenant_id + course_id` 必须唯一。
- 创建 `course_user`：`user_id` 为课程创建人、`access_role = owner`、`access_status = active`。
- 课程、知识库和创建人 `owner` 授权必须在同一事务中成功或回滚。
- 修改课程名称不自动修改知识库名称，避免覆盖管理员后续自定义名称。
- 初始化后学员仍无可用知识库；首次发布成功后才设置 `current_version_id`。

## 3. 状态设计

| 状态 | 含义 |
|---|---|
| `draft` | 草稿，可调整纳入资料 |
| `building` | 正在构建 Chunk 和索引 |
| `ready` | 构建成功，等待人工发布 |
| `published` | 当前线上版本 |
| `archived` | 历史版本，可审计和回滚 |
| `failed` | 构建失败，可查看原因并重试 |

状态字典使用 `trainmind_knowledge_base_version_status`。字典只负责展示，后端使用常量或枚举校验状态流转。

```text
draft -> building -> ready -> published -> archived
   |         |
   |         -> failed -> building
   -> archived
```

规则：

- 只有 `draft` 可以增删资料快照。
- `building` 开始后冻结资料快照。
- `ready` 必须人工发布，不自动发布。
- 发布事务同时更新新旧版本状态和 `knowledge_base.current_version_id`。
- 构建或发布失败不得影响当前发布版本。
- 回滚不直接重新激活历史版本，而是复制历史资料快照并生成新的递增版本，再走构建和发布流程。

## 4. 表结构

所有业务表统一包含：

```text
tenant_id
del_flag
create_by
create_time
update_by
update_time
remark
```

删除统一设置 `del_flag = '2'`，查询默认使用 `del_flag = '0'`，不执行物理删除。

### 4.1 knowledge_base

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `tenant_id` | 租户 ID |
| `course_id` | 课程 ID，MVP 唯一 |
| `name` | 知识库名称 |
| `description` | 描述 |
| `current_version_id` | 当前已发布版本 ID |
| `status` | `active / archived` |
| `del_flag` | 逻辑删除 |
| `create_by` | 创建人 |
| `create_time` | 创建时间 |
| `update_by` | 更新人 |
| `update_time` | 更新时间 |
| `remark` | 备注 |

### 4.2 knowledge_base_version

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `tenant_id` | 租户 ID |
| `knowledge_base_id` | 知识库 ID |
| `version_no` | 系统递增版本号 |
| `status` | 版本状态 |
| `chunk_count` | Chunk 数量 |
| `chunk_strategy_version` | 切分策略版本 |
| `retrieval_strategy_version` | 检索策略版本 |
| `embedding_index_version_id` | 向量索引版本 |
| `keyword_index_version_id` | 关键词索引版本 |
| `published_at` | 发布时间 |
| `published_by` | 发布人 |
| `del_flag` | 逻辑删除 |
| `create_by` | 创建人 |
| `create_time` | 创建时间 |
| `update_by` | 更新人 |
| `update_time` | 更新时间 |
| `remark` | 备注 |

### 4.3 knowledge_base_version_document

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `tenant_id` | 租户 ID |
| `knowledge_base_version_id` | 知识库版本 ID |
| `document_id` | 资料 ID |
| `document_version_id` | 纳入的资料版本 ID |
| `del_flag` | 逻辑删除 |
| `create_by` | 创建人 |
| `create_time` | 创建时间 |
| `update_by` | 更新人 |
| `update_time` | 更新时间 |
| `remark` | 备注 |

同一知识库版本下 `document_id` 必须唯一。只有未删除且 `status = parsed` 的资料版本可以加入草稿。资料版本被知识库发布后仍保持 `parsed`，因此可以被下一版草稿继续复用。

文档版本只表达文件生命周期，使用 `uploaded / parsing / parsed / failed / archived`。知识结构处理、整库构建和发布状态由 `knowledge_base_version.status` 表达。

表中存在且 `del_flag = '0'` 的关联即表示资料已纳入。草稿排除资料时将关联记录逻辑删除；构建、发布和草稿复制只读取 `del_flag = '0'`，不使用额外的 `include_status`。

### 4.4 course_user

MVP 使用直接课程授权，不实现班级授权。

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `tenant_id` | 租户 ID |
| `course_id` | 课程 ID |
| `user_id` | 用户 ID |
| `access_role` | `owner / teacher / student` |
| `access_status` | `active / disabled` |
| `start_at` | 授权开始时间 |
| `end_at` | 授权结束时间，可为空 |
| `del_flag` | 逻辑删除 |
| `create_by` | 创建人 |
| `create_time` | 创建时间 |
| `update_by` | 更新人 |
| `update_time` | 更新时间 |
| `remark` | 备注 |

`tenant_id + course_id + user_id` 必须唯一。课程访问时同时校验 `access_status`、授权时间范围和 `del_flag`。

主负责人规则：

- `course.owner_user_id` 是唯一主负责人的权威字段。
- 每门课程只允许一个有效 `course_user(access_role = owner)`，并且必须与 `owner_user_id` 一致。
- 创建课程时在同一事务写入两处负责人信息。
- 主负责人不能直接删除、停用或改为其他角色。
- 变更负责人必须使用“转移负责人”，同一事务更新 `course.owner_user_id`、原负责人的课程授权和新负责人的 `owner` 授权。
- 其他课程管理人员使用 `teacher`。

## 5. 业务流程

```text
创建或打开知识库草稿
-> 从 parsed 资料版本中选择快照
-> 发起构建
-> AI 服务生成 Chunk、Embedding 和关键词索引
-> 构建成功进入 ready
-> 管理员人工确认发布
-> 切换 current_version_id
-> 学员资料列表和 AI 问答读取新发布版本
```

上传或解析新资料不会自动修改现有草稿，也不会影响当前发布版本。

### 5.1 草稿快照初始化

- 首次创建草稿时，因为没有当前发布版本，资料快照为空，由管理员选择资料版本。
- 后续创建草稿时，自动复制当前发布版本的全部资料快照。
- 复制的只是 `knowledge_base_version_document` 关联关系，不复制原始文件、解析结果、Chunk 或索引。
- 管理员在复制后的草稿中替换资料版本、增加资料或排除资料。
- 创建草稿后新上传或新解析完成的资料不会自动加入草稿，必须由管理员明确选择。

### 5.2 单活动版本约束

同一知识库最多只能存在一个活动中的非历史版本，以下状态合计最多一条：

```text
draft / building / ready / failed
```

规则：

- 当前活动版本发布或归档后，才能创建下一个草稿。
- `published` 当前版本和多个 `archived` 历史版本不计入活动版本限制。
- 创建草稿时必须通过数据库唯一约束或事务锁防止并发创建。
- 前端检测到已有活动版本时直接进入该版本，不再显示“创建草稿”。

### 5.3 MVP 策略配置

MVP 的 Chunk 和检索策略由系统默认配置决定：

- 创建草稿时自动写入当前默认 `chunk_strategy_version`。
- 创建草稿时自动写入当前默认 `retrieval_strategy_version`。
- 策略版本写入知识库版本后固定，不随系统默认配置变化。
- 知识库治理页面只读展示本版本使用的策略，不提供切换控件。
- 后续完成策略评估和权限设计后，再开放按知识库版本选择策略。

### 5.4 资料变更与全量重建

已发布知识库中的某个课件发生变化时，不修改当前发布版本及其 Chunk 和索引：

1. 在资料管理中上传新的文档版本。
2. 手动解析新文档版本，等待状态进入 `parsed`。
3. 创建新知识库草稿，自动复制当前发布版本的资料快照。
4. 在草稿中将旧文档版本替换为新文档版本。
5. 保存快照并发起知识库构建。
6. 构建成功进入 `ready`，由有权限的用户人工发布。
7. 发布事务切换 `knowledge_base.current_version_id`，原发布版本进入 `archived`。

MVP 采用全量重建：

- 每个新知识库版本重新处理快照内的全部资料。
- 为新版本重新生成全部 Chunk、Embedding 和关键词索引。
- 不复制或复用历史知识库版本的 Chunk 和索引记录。
- 所有新 Chunk 和索引必须关联新的 `knowledge_base_version_id`。
- 构建和发布失败时，当前发布版本继续提供学员资料和 AI 问答服务。

增量复制未变化资料的 Chunk 属于后续性能优化，不进入 MVP。

## 6. 接口

| 接口 | 方法 | 说明 |
|---|---|---|
| `/course/{courseId}/knowledge-base` | `GET` | 查询课程知识库 |
| `/course/{courseId}/knowledge-base/versions` | `GET` | 查询版本历史 |
| `/course/{courseId}/knowledge-base/versions` | `POST` | 创建草稿版本 |
| `/course/{courseId}/knowledge-base/versions/{versionId}/documents` | `GET` | 查询资料快照 |
| `/course/{courseId}/knowledge-base/versions/{versionId}/documents` | `PUT` | 保存草稿资料快照 |
| `/course/{courseId}/knowledge-base/versions/{versionId}/build` | `POST` | 发起构建 |
| `/course/{courseId}/knowledge-base/versions/{versionId}/build-status` | `GET` | 查询构建状态 |
| `/course/{courseId}/knowledge-base/versions/{versionId}/publish` | `POST` | 人工发布 |
| `/course/{courseId}/knowledge-base/versions/{versionId}/rollback` | `POST` | 回滚到指定版本 |
| `/course/{courseId}/published-documents` | `GET` | 学员查询当前发布资料 |
| `/course/{courseId}/members` | `GET` | 查询课程成员 |
| `/course/{courseId}/members` | `POST` | 添加直接授权 |
| `/course/{courseId}/members/{courseUserId}` | `PUT` | 修改角色、状态和有效期 |
| `/course/{courseId}/members/{courseUserId}` | `DELETE` | 逻辑删除直接授权 |

AI 服务构建接口必须接收 `knowledge_base_version_id`；单文件解析接口不接收该字段。

全量构建时，Java 只向 AI 服务传递 `knowledge_base_version_id`。AI 服务使用共享 PostgreSQL 的只读权限查询 `knowledge_base_version_document`，取得该版本全部有效 `document_id / document_version_id`，然后一次性删除该版本旧 Chunk、遍历全部资料生成新 Chunk，并构建整版索引。

## 7. 权限

权限通过 RuoYi“用户 -> 角色 -> 菜单/按钮权限”模型分配，业务代码不写死角色名称。接口必须同时校验功能权限和当前课程的数据权限。

| 操作 | 权限标识 | 默认分配 |
|---|---|---|
| 查看治理状态 | `course:knowledge-base:query` | 管理员、课程讲师/负责人 |
| 创建和维护草稿 | `course:knowledge-base:edit` | 管理员、课程讲师/负责人 |
| 发起构建和重试 | `course:knowledge-base:build` | 管理员、课程讲师/负责人 |
| 发布和回滚 | `course:knowledge-base:publish` | 管理员 |
| 查看课程成员 | `course:member:list` | 管理员、课程负责人 |
| 管理课程成员 | `course:member:edit` | 管理员、课程负责人 |
| 访问当前发布资料 | 课程学习访问权限 | 管理员、课程讲师/负责人、学员 |

规则：

- 系统管理员拥有全部权限。
- 课程讲师/负责人默认不能发布和回滚；管理员可通过角色配置为其授予 `course:knowledge-base:publish`。
- 学员角色不分配任何知识库治理菜单和按钮权限。
- 具备功能权限但不在当前课程授权范围内的用户仍不能操作该课程。
- 发布和回滚权限不能由资料上传或知识库构建权限隐式获得。

MVP 使用 `course_user` 直接授权课程访问，不实现班级授权。只有存在有效 `course_user` 记录的学员，才能访问课程当前发布资料和 AI 问答。

## 8. 页面设计

### 8.1 菜单归属与入口

MVP 不新增独立的“知识库管理”菜单。知识库归属于课程，统一从课程详情进入：

```text
课程管理
└── 课程列表
    └── 课程详情
```

原因：

- MVP 已确定一门课程对应一个知识库。
- 知识库权限必须同时受课程数据权限约束。
- 资料、模块和知识库版本需要在同一课程上下文中操作。

课程详情包含 5 个一级 Tab：

```text
基本信息 / 模块管理 / 资料管理 / 知识库治理 / 课程成员
```

“课程成员”只维护 `course_user` 直接授权，不包含班级。班级和选课关系留待后续实施。

“知识库治理”只对管理员和具备相应权限的课程讲师/负责人显示。学员端不展示该 Tab。

### 8.2 知识库治理子 Tab

知识库治理包含 3 个子 Tab：

| 子 Tab | 作用 |
|---|---|
| 草稿资料 | 选择资料版本、保存草稿、发起构建 |
| 构建任务 | 查看构建进度、子任务、失败原因和重试 |
| 版本历史 | 查看当前发布版本、历史快照和创建回滚版本 |

知识库名称、当前发布版本、活动版本状态和主操作位于固定顶部状态区，切换子 Tab 时保持可见。

### 8.3 页面整体结构

```text
知识库治理
├── 顶部状态栏
│   ├── 知识库名称
│   ├── 当前发布版本
│   ├── 发布时间
│   ├── 当前草稿/构建版本
│   └── 创建草稿、构建、发布等主操作
├── 子 Tab：草稿资料
│   ├── 左侧模块筛选
│   └── 右侧资料版本快照
├── 子 Tab：构建任务
└── 子 Tab：版本历史
```

页面不使用嵌套卡片。顶部为紧凑状态栏，主体使用子 Tab 组织“草稿资料”“构建任务”“版本历史”。

### 8.4 顶部状态栏

展示：

- 知识库名称。
- 知识库状态。
- 当前发布版本号；尚未发布时显示“暂无发布版本”。
- 当前发布版本的发布时间和发布人。
- 当前操作版本号及其 `draft / building / ready / failed` 状态。

操作：

- `创建草稿`：没有可编辑草稿时显示。
- `保存草稿`：草稿资料发生变化后启用。
- `发起构建`：草稿已保存且至少包含一个资料版本时启用。
- `重新构建`：构建失败时显示。
- `发布`：仅 `ready` 状态且拥有发布权限时启用。
- 更多菜单：归档草稿、查看当前发布快照。

发布按钮必须弹出确认对话框，明确显示即将发布的版本、资料数量以及将被替换的当前版本。

### 8.5 草稿资料 Tab

左侧模块筛选：

```text
全部资料
课程公共资料
M1
M2
...
```

右侧工具栏：

- 资料名称搜索。
- 文件类型筛选。
- “仅看已选择”开关。
- “使用最新可用版本”批量操作。

资料快照表格：

| 列 | 说明 |
|---|---|
| 选择 | 是否纳入当前草稿 |
| 资料名称 | `course_document.title` |
| 所属模块 | 公共资料或模块名称 |
| 纳入版本 | 可选择 `parsed / published` 的具体版本 |
| 文件类型 | PDF、DOCX、PPTX、XLSX |
| 文件大小 | 当前所选版本大小 |
| 版本状态 | 使用资料版本状态字典 |
| 更新时间 | 版本上传时间 |
| 操作 | 下载、查看版本历史 |

切换资料版本只修改草稿快照，不修改 `course_document.latest_version_id`。已进入 `building` 后，选择框和版本选择器全部只读。

### 8.6 构建任务 Tab

展示当前版本的构建摘要：

- 总体状态、当前步骤、进度百分比。
- 纳入资料数量、成功数量、失败数量。
- Chunk 数量。
- 向量索引和关键词索引状态。
- 开始时间、结束时间、耗时。
- 失败错误码和错误原因。

任务列表字段：

| 列 | 说明 |
|---|---|
| 任务类型 | 解析、结构化、Chunk、Embedding、关键词索引、整库构建 |
| 状态 | pending、running、success、failed、cancelled |
| 当前步骤 | Worker 当前步骤 |
| 进度 | 百分比 |
| 重试次数 | 已重试次数 |
| 开始/结束时间 | 任务执行时间 |
| 操作 | 查看错误、重试 |

### 8.7 版本历史 Tab

版本列表字段：

| 列 | 说明 |
|---|---|
| 版本号 | 系统生成 |
| 状态 | draft、building、ready、published、archived、failed |
| 资料数量 | 快照资料数 |
| Chunk 数量 | 构建后的 Chunk 数 |
| 构建时间 | 最近构建完成时间 |
| 发布时间 | 实际发布时间 |
| 发布人 | 发布操作人 |
| 操作 | 查看快照、查看任务、回滚 |

点击“查看快照”打开抽屉，展示该版本固定的资料与文档版本，不允许编辑。

“回滚”仅对可回滚的历史版本显示，并弹出二次确认。回滚完成后页面刷新当前发布版本和历史版本状态。

回滚操作实际执行：

1. 校验目标历史版本完整且可回滚。
2. 复制目标版本的资料快照，创建新的 `version_no`。
3. 新版本进入 `draft`，允许管理员检查快照。
4. 管理员重新发起构建。
5. 构建进入 `ready` 后人工发布。

历史版本本身保持 `archived` 且不可修改。页面按钮文案使用“基于此版本创建回滚版本”，避免用户误以为点击后会立即切换线上版本。

### 8.8 页面状态

- 无知识库：显示初始化操作，不显示空表格。
- 有知识库但无版本：引导创建首个草稿。
- 构建中：页面允许查看，禁止修改草稿和重复构建。
- 构建失败：突出失败原因并提供重试。
- 待发布：突出 `ready` 状态和发布操作。
- 无发布版本：学员资料和问答入口均不可用。
- 接口异常：保留已加载数据并提供重试，不把状态误显示为空。

### 8.9 尚无发布版本时的学员端

课程尚无 `current_version_id` 时：

- 学员仍可进入已获得访问权限的课程。
- 课程资料列表返回空集合。
- AI 问答入口禁用。
- 页面显示“课程资料尚未发布”。
- 不允许回退读取最新上传、最新解析或草稿版本资料。
- 管理端不受影响，仍可上传、解析、维护草稿和发起构建。

### 8.10 课程成员 Tab

课程成员用于维护 `course_user` 直接授权。

页面工具栏：

- 用户姓名或账号搜索。
- 访问角色筛选。
- 授权状态筛选。
- 添加成员。

列表字段：

| 列 | 说明 |
|---|---|
| 用户 | 用户姓名和账号 |
| 访问角色 | `owner / teacher / student` |
| 授权状态 | `active / disabled` |
| 生效时间 | `start_at` |
| 失效时间 | `end_at` |
| 创建人 | 授权操作人 |
| 操作 | 编辑授权、启停、移除 |

添加成员时从系统用户中选择，不重复创建用户。保存前校验同一课程不能重复授权同一用户。

主负责人行不显示普通的停用、移除和角色修改操作，只显示“转移负责人”。转移对话框选择新的系统用户并明确提示负责人权限将发生变化。

## 9. 删除和归档

- 已被保留中的知识库版本引用的资料版本不得物理删除。
- 当前发布版本不得删除，只能由新版本发布替换后归档。
- 历史版本保留元数据、资料快照和审计所需数据，清理由独立保留策略控制。
- 逻辑删除、对象清理、解析结果和索引清理前必须完成引用检查。

## 10. 验收标准

- 解析成功不会让资料自动对学员可见。
- 草稿只能选择解析成功且仍有效的资料版本。
- 构建失败时当前发布版本继续服务。
- `ready` 版本必须人工发布后才对学员生效。
- 发布后学员资料和 AI 问答读取同一个 `current_version_id`。
- 上传新资料版本期间，旧发布资料持续可见。
- 可以查看历史版本快照并执行回滚。
- 已发布快照不会被资料逻辑删除破坏。
