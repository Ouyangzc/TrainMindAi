# 企业级知识助教系统落地方案

## 1. 文档目的

本文档用于指导后续真实企业级知识助教系统的产品设计、架构设计、开发拆分和分阶段落地。

当前仓库中的 Streamlit 原型已经跑通了基础链路：

```text
PPT课件 -> Markdown知识库 -> Chunk切分 -> Embedding入库 -> RAG检索 -> AI问答/出题 -> 错题诊断 -> 知识点重讲
```

但该原型只适合学习和演示。企业级项目需要补齐用户、课程、知识库治理、权限、题库、作答记录、学习画像、日志审计、评估体系和可运维能力。

本文档的目标不是描述当前代码，而是定义后续要建设的目标系统。

## 2. 系统定位

系统定位为：

```text
面向企业培训、职业课程、内部知识学习场景的 AI 知识助教平台。
```

核心价值不是“让用户问文档”，而是：

```text
让学员更快学会知识；
让教师更快备课和诊断问题；
让企业知道员工学会了什么、没学会什么、哪些知识点存在风险。
```

## 3. 业务目标

### 3.1 学员侧目标

- 支持围绕课程资料进行自然语言提问。
- 回答必须带有课件或资料来源。
- 支持按章节、知识点进行练习。
- 支持错题记录、错因分析和知识点重讲。
- 支持个人学习进度和薄弱知识点追踪。

### 3.2 教师侧目标

- 支持上传课程资料并维护知识点结构。
- 支持基于知识点自动生成题目。
- 支持审核、编辑、发布 AI 生成题目。
- 支持查看班级薄弱知识点和高频错题。
- 支持生成复习建议和教学改进建议。

### 3.3 管理侧目标

- 查看企业、部门、班级的培训完成情况。
- 查看课程掌握度、考试通过率和学习风险。
- 识别高频问题、高风险知识点和资料质量问题。
- 支持数据导出、审计和运营分析。

## 4. 当前原型诊断

### 4.1 已具备能力

- PPT 文本抽取。
- Markdown 中间知识库。
- 按标题层级切分 chunk。
- ChromaDB 向量存储。
- RAG 问答。
- AI 出题。
- 错题诊断。
- 知识点重讲。
- 本地演示启动和便携版打包。

### 4.2 当前不足

- 没有用户、角色、组织、班级、课程权限。
- 没有正式业务数据库。
- 没有资料上传、审核、发布和版本管理。
- 向量库更新依赖人工操作。
- 没有混合检索、rerank、检索置信度判断。
- 没有问答日志、检索日志、模型调用日志。
- 没有标准问答集和 RAG 评估体系。
- AI 生成题目没有审核流。
- 错题和学习记录没有持久化。
- 不能支持多人并发和企业级部署。

## 5. 建设原则

### 5.1 以知识点为核心

系统不要只围绕“文档”建模，而要围绕“课程知识点”建模。

推荐主线：

```text
课程 -> 章节 -> 知识点 -> 资料片段 -> 题目 -> 作答 -> 掌握度
```

只有知识点稳定，才能支撑出题、诊断、错题本、掌握度和教学看板。

### 5.2 RAG 必须可评估

企业级 RAG 不能只看“能否回答”，必须能回答：

- 检索到了哪些资料？
- 命中分数是多少？
- 引用是否正确？
- 回答是否基于资料？
- 改了 chunk 策略后效果变好还是变差？

因此要尽早建设检索日志和评估集。

### 5.3 AI 结果必须可审核

AI 生成的题目、解析、诊断建议不能直接作为正式内容发布。

需要至少支持：

```text
AI生成 -> 草稿 -> 人工审核 -> 发布 -> 使用
```

### 5.4 业务系统和 AI 能力分层

推荐分层：

```text
业务系统负责用户、课程、权限、记录、报表。
AI 服务负责解析、切片、检索、生成、评估。
```

不要把权限、课程规则、学习记录全部塞进 RAG 函数里。

## 6. 目标用户与角色

| 角色 | 主要职责 | 核心功能 |
| --- | --- | --- |
| 学员 | 学习、提问、练习、复习 | 问答、练习、错题本、重讲、学习进度 |
| 教师 | 备课、出题、诊断、教学改进 | 资料管理、题目生成、题库审核、班级分析 |
| 培训管理员 | 课程和班级运营 | 班级管理、课程分配、考试安排、学习报表 |
| 知识库管理员 | 资料治理 | 文档上传、解析审核、知识点维护、版本发布 |
| 系统管理员 | 系统配置和安全 | 用户权限、模型配置、日志审计、系统监控 |
| 企业管理者 | 看整体培训效果 | 完成率、通过率、掌握度、风险知识点 |

§3 的三类业务目标（学员侧 / 教师侧 / 管理侧）与上表的对应关系为：学员侧对应“学员”；教师侧对应“教师”；管理侧由“培训管理员、知识库管理员、系统管理员、企业管理者”四类角色按职责细分承担。MVP 阶段可先合并管理侧角色，仅区分学员、教师、管理员三类，后续再按上表拆分。

### 7.2 推荐技术选型

## 8. 核心业务模块

### 8.1 用户与权限模块

主要能力：

- 企业、部门、用户、角色管理。
- 学员、教师、管理员角色划分。
- 课程访问权限。
- 知识库访问权限。
- 操作审计。

关键点：

- 问答检索前必须做权限过滤。
- MVP 阶段可先支持班级授权和用户直接授权。
- 教师只能查看自己负责班级或课程的数据。
- 管理者只能查看授权范围内的统计数据。

### 8.2 课程管理模块

主要能力：

- 课程创建、编辑、上下架。
- 章节维护。
- 知识点维护。
- 课程资料绑定。
- 课程班级分配。

推荐结构：

```text
Course
  -> Chapter
      -> KnowledgePoint
          -> DocumentChunk
          -> Question
          -> MasteryRecord
```

### 8.3 知识库管理模块

主要能力：

- 上传资料。
- 资料解析。
- Markdown 中间层预览。
- 章节和知识点基础维护。
- 人工修正知识点。
- 生成 chunk。
- 向量化。
- 发布知识库版本。
- 回滚历史版本。

知识库不是文件夹，也不是向量库，而是课程下可发布、可问答的知识集合。MVP 阶段建议先采用“一门课程一个知识库”的模型，后续再扩展为同一课程下多个知识库。

知识库版本是一次可问答知识快照。一次版本发布应固定：

- 本次使用的资料版本。
- 本次生成的 chunk。
- 本次构建的向量索引。
- 本次构建的全文索引。
- 发布后的问答和检索日志使用的知识库版本。

问答不能直接读取“最新资料”或“最新 chunk”，必须读取课程当前发布的 `knowledge_base_version`。这样资料更新、索引重建、版本回滚和问答复盘才有稳定依据。

资料状态（作用于 `document_version`，描述单个资料版本从上传到发布的生命周期，用统一的 `status` 字段表示）：

```text
uploaded
parsing
parsed
structuring
reviewing
published
failed
archived
```

知识库版本状态：

```text
draft
building
ready
published
archived
failed
```

### 8.4 RAG 问答模块

企业级问答流程：

```text
用户提问
-> 用户权限和课程范围判断
-> 获取课程当前发布的知识库版本
-> 查询改写
-> 向量检索
-> 关键词检索
-> 元数据过滤
-> 候选结果合并
-> Rerank（阶段 4 可选增强）
-> 置信度判断
-> Prompt 组装
-> LLM 生成
-> 引用校验
-> 返回答案
-> 记录问答日志和检索日志
```

必须支持：

- 按课程范围问答。
- 按知识库版本问答。
- 回答带引用。
- 资料不足时拒答。
- 用户反馈“有用 / 无用”。
- 管理员查看低质量问答。

### 8.5 题库与测评模块

主要能力：

- AI 生成题目。
- 人工审核题目。
- 题库管理。
- 组卷。
- 练习。
- 考试。
- 自动判分。
- 简答题 AI 辅助评分。

题目状态：

```text
ai_draft
pending_review
approved
rejected
published
disabled
```

题型：

- 单选题。
- 多选题。
- 判断题。
- 填空题。
- 简答题。
- 案例题。

### 8.6 错题诊断模块

主要能力：

- 自动记录错题。
- 关联错题到知识点。
- 识别错因。
- 推荐复习资料。
- 生成类似练习。
- 支持个人错题本。

错因分类建议：

```text
concept_missing：概念未掌握
memory_error：记忆错误
application_error：应用错误
careless_error：审题或粗心错误
unknown：无法判断
```

### 8.7 学习画像模块

主要能力：

- 学员知识点掌握度。
- 学员学习进度。
- 学员错题分布。
- 班级薄弱点。
- 课程掌握度热力图。
- 学习建议。

掌握度计算输入：

- 问答记录。
- 练习记录。
- 考试记录。
- 错题记录。
- 复习完成记录。

### 8.8 管理看板模块

主要指标：

- 培训完成率。
- 考试通过率。
- 平均掌握度。
- 高频提问。
- 高频错题。
- 高风险知识点。
- 课程资料质量问题。
- AI 回答满意度。

## 9. AI 能力设计

### 9.1 文档解析

支持格式：

- PPT。
- PDF。
- Word。
- Excel。
- Markdown。
- HTML。
- TXT。

输出标准结构：

```json
{
  "document_id": "doc_001",
  "version_id": "ver_001",
  "pages": [
    {
      "page_number": 1,
      "title": "章节标题",
      "text": "页面文本",
      "tables": [],
      "images": [],
      "metadata": {}
    }
  ]
}
```

### 9.2 知识结构化

目标是把原始资料变成稳定的课程知识结构：

```text
课程标题
章节标题
知识点标题
知识点说明
来源页码
考试要素
关键概念
```

当前原型中“PPT -> Markdown -> 标题层级 chunk”的路径可以继续保留，但企业级系统中 Markdown 应成为可审核、可版本化的中间资产。

### 9.3 Chunk 策略

推荐策略：

- 优先按知识点标题切分。
- 同一知识点下多页合并。
- 超长知识点按段落或语义边界拆分。
- 每个 chunk 必须保留课程、章节、知识点、资料版本、页码。
- chunk 不应只保存纯文本，应保存元数据。

chunk 元数据建议：

```json
{
  "course_id": "course_001",
  "chapter_id": "chapter_001",
  "knowledge_point_id": "kp_001",
  "document_id": "doc_001",
  "document_version_id": "ver_001",
  "source_file": "M1.1.pptx",
  "page_start": 3,
  "page_end": 5,
  "chunk_type": "knowledge_point",
  "published_version": "kb_20260615"
}
```

其中 `published_version` 为人类可读的版本标识，便于检索日志和导出阅读；落库时以 `knowledge_chunk.knowledge_base_version_id`（外键）为权威关联，二者一一对应，避免实现时产生歧义。

Chunk 策略需要版本化，不能只记录最终 chunk 文本。每次调整切分规则后，应能追溯本次知识库版本使用了哪一种策略，以及效果变化来自资料、模型还是 chunk 策略。

建议记录：

```text
chunk_strategy_version
chunk_method              // title / knowledge_point / semantic / fixed_size
chunk_size
chunk_overlap
merge_rule
split_rule
metadata_rule
created_at
```

MVP 阶段优先采用“标题层级 + 知识点合并”的策略。固定大小分块只适合作为兜底方案，不建议作为企业知识库的主策略；语义分块和 ColBERT 类动态窗口可放到阶段 4 作为召回优化增强。

### 9.4 检索策略

企业级不建议只做向量检索。推荐：

```text
向量检索 top 50
+ BM25 top 50
+ 元数据过滤
+ 去重合并
+ Rerank top 10
+ Prompt top 5
```

MVP 阶段可以暂不引入 Rerank 服务，也不强制引入 OpenSearch / Elasticsearch。第一版至少应实现“轻量混合检索”：

```text
向量检索 top 20
+ PostgreSQL 全文检索 top 20
+ 元数据过滤
+ 去重合并
+ 加权排序 top 5
+ Prompt 组装
```

这样既能控制阶段 1 的复杂度，又能保证真实资料下的召回质量、引用来源和检索日志具备演示可信度。

选择轻量混合检索的原因：

- 企业课件、制度、SOP、设备型号、法规条款、缩写和专有名词较多，单纯向量检索容易语义相近但关键词不准。
- 关键词检索对精确词、标题词、编号、页码、术语命中更稳定，适合作为“必须命中原文证据”的召回通道。
- MVP 用 PostgreSQL 全文检索即可完成关键词召回，不需要为了早期演示和试点额外维护搜索引擎。
- 检索日志里保留关键词分数、向量分数和最终排序，便于后续评估是否需要升级到标准 BM25。

生产环境再引入 OpenSearch / Elasticsearch 的原因：

- 它们原生支持倒排索引和 BM25 排序，不需要业务系统自己实现全文检索引擎。
- 支持按课程、知识库版本、文档状态、资料类型等字段过滤，适合和权限过滤、版本过滤一起使用。
- 支持中文分词、同义词、字段权重和高亮，后续可以逐步优化资料检索效果。
- 适合多课程、大资料量、高并发和需要持续调参的生产场景。

### 9.5 查询改写和召回优化

用户提问往往不是资料里的原文表达，企业资料中还会大量出现缩写、设备型号、制度编号、岗位术语和内部简称。查询改写应作为检索前的标准步骤，而不是只依赖 LLM 最终回答。

查询改写建议输出：

```text
raw_query
normalized_query       // 规范化后的用户问题
keyword_query          // 关键词检索使用
semantic_query         // 向量检索使用
expanded_terms         // 同义词、缩写、专业术语扩展
course_scope_hint      // 可能关联的课程、章节或知识点
rewrite_reason
```

召回优化建议按以下顺序推进：

```text
先检查 chunk 是否切对
-> 再检查关键词和向量召回是否互补
-> 再检查查询改写是否覆盖用户表达差异
-> 最后再引入 Rerank 或 LLM 重排序
```

不要一开始就把召回问题归因到模型生成。企业级 RAG 的多数问题来自“没有召回到正确证据”或“正确证据排得太靠后”。因此每次优化都应记录：

- 使用的 chunk 策略版本。
- 向量检索 topK 和关键词检索 topK。
- 向量分数、关键词分数、元数据加权分数和最终分数。
- 是否命中期望知识点、期望文档和期望页码。
- Prompt 中实际使用了哪些 chunk。

阶段 1 可先采用加权排序：

```text
final_score = vector_score * 0.6 + keyword_score * 0.3 + metadata_score * 0.1
```

具体权重不要写死在代码里，应作为可配置项并写入检索日志。阶段 4 再引入 BGE-Reranker、ColBERT 或 LLM Re-ranking，对 top 20 到 top 50 的候选片段进行精排序。

### 9.6 资料不足和拒答策略

系统必须允许拒答，且拒答要有可解释原因。建议触发条件包括：

- 检索结果低于最低分数阈值。
- topK 结果没有命中课程当前发布的知识库版本。
- 命中片段与问题主题不一致。
- 命中片段只有泛化描述，缺少可引用证据。
- 命中知识点为空或置信度过低。
- 用户问题超出课程授权范围。

拒答结果也要写入问答日志和检索日志，便于管理员判断是资料缺失、chunk 策略问题、检索策略问题，还是用户确实问了课程外问题。

过滤条件：

- 用户权限。
- 课程范围。
- 知识库版本。
- 文档状态。
- 语言。
- 资料类型。

### 9.7 Prompt 设计

不同场景应拆分 prompt：

- 问答 prompt。
- 出题 prompt。
- 错题诊断 prompt。
- 知识点重讲 prompt。
- 简答题评分 prompt。
- 资料质量检查 prompt。

所有 prompt 都应版本化：

```text
prompt_code
prompt_version
prompt_content
created_by
enabled
```

### 9.8 输出校验

LLM 输出必须经过校验：

- JSON schema 校验。
- 题目选项数量校验。
- 答案格式校验。
- 引用来源校验。
- 敏感内容校验。
- 资料外内容检测。

失败后策略：

```text
重试一次
-> 降级为人工处理
-> 记录失败原因
```

## 12. 关键流程

### 12.1 资料入库流程

```text
上传资料
-> 创建 document 和 document_version
-> 写入对象存储
-> 创建 knowledge_base_version 草稿
-> 绑定本次使用的 document_version
-> 创建 kb_build_task
-> 文档解析
-> 生成 Markdown 中间结果
-> 识别章节和知识点                （阶段 1 为人工维护；自动识别属阶段 4 增强）
-> 人工审核知识结构               （阶段 1 仅做基础维护；正式审核流属阶段 4 增强）
-> 生成 chunk
-> 生成 embedding
-> 写入向量库
-> 创建向量索引版本和全文索引版本
-> knowledge_base_version 状态变为 ready
-> 人工确认发布
-> 发布知识库版本
-> 更新 knowledge_base.current_version_id
```

上图为完整目标流程。阶段 1（MVP）的简化路径为：上传 -> 解析 -> 生成 Markdown -> 人工维护章节/知识点 -> 生成 chunk 和 embedding -> 构建索引 -> 发布版本。其中“知识点自动识别”和“知识结构审核流”不在阶段 1 范围内（见 §17 阶段 1 的“不做”），阶段 1 只提供知识点的人工基础维护。

解析、结构化、chunk、embedding、全文索引和重建都应写入 `kb_build_task`。管理员至少要能查看任务状态、失败原因并重试失败任务。

MVP 阶段不强制引入独立消息队列。后端或 AI 服务可以启动轻量后台 Worker，轮询 `kb_build_task` 或使用 Redis 简单队列执行构建任务。进入多课程、大资料量、并发构建后，再升级为独立 Worker 集群和 RabbitMQ / Kafka。

知识库重建不能直接覆盖当前线上版本。正确流程是：

```text
当前 published v3 正常服务
-> 创建 v4 draft/building
-> 执行 v4 构建任务
-> v4 构建成功后进入 ready
-> 人工确认发布
-> current_version_id 从 v3 切换到 v4
-> v3 archived，可回滚
```

如果 v4 构建失败，v3 必须继续服务，管理员通过失败任务查看原因并重试 v4 构建。

### 12.2 问答流程

```text
用户提问
-> 创建 qa_message
-> 判断课程权限
-> 获取课程当前发布的 knowledge_base_version

-> 混合检索
-> Rerank（阶段 4 可选增强）
-> 记录 qa_retrieval_log
-> 判断资料是否足够
-> 组装 prompt
-> 调用模型
-> 引用校验
-> 保存回答
-> 返回答案和来源
```

### 12.3 出题流程

```text

-> 检索相关 chunk
-> 生成题目 JSON
-> Schema 校验
-> 选项和答案格式化
-> 保存为 ai_draft
-> 教师审核
-> 发布到题库
```

### 12.4 作答与诊断流程


## 13. 部署架构

### 13.1 MVP 部署

适合第一阶段：

```text
前端
后端业务服务
AI 服务
AI Worker（MVP 可与 AI 服务同进程部署）
PostgreSQL
Redis
MinIO
Qdrant 或 pgvector
```

MVP 部署不是生产高可用架构，而是用于验证核心业务闭环和 RAG 效果的最小可用环境。建议使用 Docker Compose 或同等方式部署：

- PostgreSQL 存储用户、课程、知识库版本、问答日志和学习记录。
- MinIO 存储原始资料、解析后的 Markdown 和附件。
- Qdrant 或 pgvector 存储向量索引。
- PostgreSQL 全文检索用于轻量关键词召回。
- AI Worker 执行资料解析、chunk、embedding、全文索引和知识库版本重建任务；MVP 可先与 AI 服务同进程部署。
- Redis 用于会话、任务状态、限流或热点配置。

该部署不引入独立搜索引擎和独立消息队列，避免 MVP 过重。进入多课程、大资料量或生产运行后，再将关键词召回迁移到 OpenSearch / Elasticsearch，并启用标准 BM25、字段权重、同义词和高亮；同时将 AI Worker 拆成独立任务集群，并引入 RabbitMQ / Kafka。

## 14. 安全与合规

MVP 安全底线：

- 登录认证。
- RBAC 权限。
- 数据权限。
- 资料访问控制。
- 文件类型校验。
- 密钥独立管理。
- 模型调用日志不记录密钥。
- prompt 和回答日志按企业要求控制可见范围。
- 上传资料、解析结果、向量索引和问答日志必须有删除或归档策略。
- prompt、检索片段、回答日志和评估结果必须按角色控制可见范围。

生产增强能力：

- 模型调用审计。
- 操作审计。
- 敏感词过滤。
- 文件病毒扫描。
- API 限流。
- 敏感数据脱敏。
- 管理员操作留痕和审计导出。

模型调用日志要避免保存敏感密钥，prompt 中如果包含敏感数据，需要根据企业要求做脱敏或访问控制。

日志和资料保留建议：

- 问答日志、检索日志、模型调用日志应配置保留周期，默认不无限期保存。
- 上传资料删除时，应同步处理对象存储文件、解析结果、chunk、向量索引和全文索引。
- 已发布知识库版本被归档后，应保留可审计的版本元数据，但可按企业策略清理大文件和索引数据。
- 管理员导出的日志和评估报告应记录导出人、导出时间和筛选条件。

## 16. RAG 评估体系

RAG 评估不应等到平台完全成型后再做。阶段 1 至少要具备轻量离线评估能力，用于判断 chunk、查询改写、混合检索权重和 prompt 调整是否真的提升效果。

### 16.1 标准评估集

每门课程应维护一套标准评估集：

```text
问题
标准答案
期望命中知识点
期望来源文档
期望页码
期望关键词
是否允许拒答
难度
问题类型
```

评估集应覆盖：

- 章节内事实问答。
- 跨页综合问答。
- 专有名词、缩写、编号和设备型号。
- 容易混淆的相似知识点。
- 资料中没有答案、应该拒答的问题。
- 用户口语化表达但资料使用正式术语的问题。

### 16.2 评估指标

检索指标：

- Recall@K。
- MRR。
- 命中知识点准确率。
- 来源页码准确率。

回答指标：

- 答案正确率。
- 引用准确率。
- 资料外生成率。
- 拒答准确率。

运营指标：

- 用户满意率。
- 追问率。
- 转人工率。
- 高频失败问题。

### 16.3 轻量离线评估流程

每次修改 chunk 策略、查询改写、检索权重、Rerank 或 prompt 后，至少执行一次离线评估：

```text
选择课程评估集
-> 使用指定 knowledge_base_version
-> 执行查询改写
-> 执行混合检索
-> 记录 topK 命中结果
-> 生成回答
-> 计算检索指标和回答指标
-> 输出本次评估报告
-> 与上一次基线对比
```

评估报告至少包含：

- 本次使用的知识库版本。
- 本次使用的 chunk 策略版本。
- 本次使用的检索策略和权重。
- Recall@K、MRR、引用准确率和拒答准确率。
- 失败问题清单。
- 每个失败问题的原因归类：chunk 问题、查询改写问题、关键词召回问题、向量召回问题、排序问题、资料缺失、生成问题。

## 17. 分阶段落地计划

### 阶段 0：原型沉淀

目标：

```text
把当前 demo 的有效经验沉淀为可复用 AI 能力。
```

交付物：

- 当前流程说明文档。
- 企业级落地方案文档。
- RAG 原型代码。
- 基础测试。

验收标准：

- 能说明 PPT 到问答的完整链路。
- 能演示提问、出题、诊断、重讲。
- 能明确 demo 和企业级系统的差距。

### 阶段 1：企业级 MVP

目标：

```text
做出可多人使用、可上传资料、可问答、可记录日志的最小系统。
```

范围：

- 登录和角色。
- 课程管理。
- 资料上传。
- 文档解析任务。
- 章节和知识点基础维护。
- 向量化任务。
- PostgreSQL 全文索引。
- 轻量混合检索。
- 查询改写。
- RAG 问答。
- 回答引用。
- 问答日志。
- 检索日志。
- 轻量 RAG 离线评估。

不做：

- 复杂考试系统。
- 多租户计费。
- 高级学习画像。
- 复杂 Agent。
- 知识点自动识别和审核流。
- 独立 Rerank 服务。

验收标准：

- 管理员可以上传一门课程资料并发布知识库。
- 学员可以在授权课程内提问。
- 回答带来源。
- 管理员可以查看检索日志。
- 检索日志可以区分向量召回、关键词召回和最终排序结果。
- 资料不足时可以拒答并记录拒答原因。
- 可以用最小评估集离线验证 Recall@K、引用准确率和拒答准确率。
- 资料变更后可以重建知识库版本。

### 阶段 2：题库和学习闭环

目标：

```text
从问答系统升级为学习助教系统。
```

范围：

- AI 出题。
- 题目审核。
- 正式题库。
- 练习。
- 作答记录。
- 错题本。
- 知识点掌握度。
- 个人复习建议。

验收标准：

- 教师可以基于知识点生成题目并审核发布。
- 学员可以练习并产生作答记录。
- 系统可以自动记录错题。
- 系统可以展示个人薄弱知识点。

### 阶段 3：教师和管理看板

目标：

```text
支撑教学运营和企业培训管理。
```

范围：

- 班级管理。
- 课程分配。
- 班级学习进度。
- 班级薄弱点。
- 高频错题。
- 课程掌握度。
- 培训完成率。
- 报表导出。

验收标准：

- 教师可以查看班级薄弱知识点。
- 管理者可以查看部门培训完成率和通过率。
- 系统可以导出学习分析报表。

### 阶段 4：企业级增强

目标：

```text
提升系统稳定性、安全性、可运营性和 AI 效果。
```

范围：

- Rerank。
- RAG 评估平台。
- Prompt 版本管理。
- 模型网关。
- 审计日志。
- 监控告警。
- 灰度发布。
- 多租户。

验收标准：

- RAG 效果可量化评估。
- Prompt 和索引版本可回滚。
- 关键任务失败可定位。
- 系统具备生产级监控和审计。

## 18. 里程碑建议

| 里程碑 | 周期 | 目标 |
| --- | --- | --- |
| M0 | 1 周 | 梳理当前原型和目标方案 |
| M1 | 2-4 周 | 企业级 MVP：课程、资料、知识库、轻量混合检索、问答、日志 |
| M2 | 3-5 周 | 题库、作答、错题、掌握度 |
| M3 | 2-4 周 | 教师看板、管理看板、报表 |
| M4 | 持续 | Rerank、评估体系、监控、安全 |

上表周期为乐观估算，假设团队对技术栈熟悉且资料质量可控。M1 需覆盖登录、RBAC、课程、解析任务、轻量混合检索、查询改写、问答、问答/检索双日志、离线评估和版本重建（见 §21 验收清单），工作量偏紧，实际排期建议预留缓冲或按需拆为 M1a（课程+资料+知识库版本）与 M1b（混合检索+问答+日志+评估）两个子里程碑。

## 19. 实施优先级

优先做：

1. 课程、章节、知识点数据模型。
2. 文档上传和解析任务。
3. 知识库版本和 chunk 表。
4. 向量索引和 PostgreSQL 全文索引。
5. 轻量混合检索和 RAG 问答 API。
6. 检索日志和模型调用日志。
7. AI 出题和题目审核。
8. 作答记录和错题本。
9. 知识点掌握度。

后置做：

1. 复杂 Agent 编排。
2. 知识图谱。
3. 多模型自动路由。
4. 高级推荐算法。
5. 多租户计费。

## 20. 主要风险

### 20.1 RAG 效果不可控

风险：

```text
检索不到、检索错、回答脱离资料。
```

应对：

- 建设评估集。
- 记录检索日志。
- 阶段 1 引入轻量混合检索，避免 MVP 只依赖向量召回。
- 阶段 4 引入 Rerank 和评估平台，持续优化检索排序效果。
- 资料不足时允许拒答。

### 20.2 资料质量差

风险：

```text
PPT/PDF 抽取文本混乱，导致知识点识别和检索效果下降。
```

应对：

- 保留 Markdown 中间层。
- 支持人工审核和修正。
- 建立资料解析质量检查。

### 20.3 AI 生成内容不稳定

风险：

```text
题目格式错误、答案错误、解析不一致。
```

应对：

- JSON schema 校验。
- 题目审核流。
- 输出重试。
- 来源一致性校验。

### 20.4 业务闭环不足

风险：

```text
系统停留在问答工具，不能体现企业培训价值。
```

应对：

- 尽早建设作答记录。
- 尽早关联知识点掌握度。
- 面向教师和管理者建设看板。

## 21. MVP 验收清单

第一版企业级 MVP 完成时，应满足：

- 支持用户登录。
- 支持基础 RBAC 和课程数据权限。
- 支持课程创建。
- 支持章节和知识点维护。
- 支持上传课程资料。
- 支持上传文件类型校验。
- 支持资料解析任务。
- 支持生成 chunk 和 embedding。
- 支持记录 chunk 策略版本。
- 支持 PostgreSQL 全文索引。
- 支持关键词检索 + 向量检索的轻量混合检索。
- 支持基础查询改写。
- 支持知识库版本发布。
- 问答必须读取当前发布的 `knowledge_base_version`，不能读取草稿或构建中的版本。
- 支持授权学员提问。
- 回答带引用来源。
- 资料不足时支持拒答并记录原因。
- 支持记录问答日志。
- 支持记录检索日志。
- 检索日志记录向量分数、关键词分数、最终分数、检索通道和使用的 chunk。
- 支持每门课程维护最小 RAG 评估集。
- 支持离线评估 Recall@K、引用准确率和拒答准确率。
- 支持管理员查看失败日志。
- 支持查看知识库构建任务状态。
- 支持失败构建任务重试。
- 模型调用日志不记录密钥。
- 支持重建知识库。

不满足以上清单，不建议进入题库和学习画像阶段。

## 22. 与当前原型的关系

当前原型可以作为三个方面的基础：

1. AI 流程学习样例。
2. RAG 能力验证样例。
3. 未来 AI 服务中部分模块的参考实现。

可以复用的思想：

- PPT 直接解析文本。
- Markdown 作为中间知识层。
- 按标题和知识点进行 chunk。
- 回答带引用来源。
- 出题后先规范化题型和答案。

不建议直接复用为生产架构的部分：

- Streamlit 单体页面。
- session_state 作为业务状态。
- 本地 Chroma 作为唯一向量库。
- 手动点击式知识库初始化。
- 无用户和权限的运行模式。
- AI 输出直接展示或使用。

## 23. 结论

企业级知识助教系统的主线应是：

```text
以课程知识点为核心数据模型；
以 RAG 问答作为学习入口；
以题库测评和错题诊断形成学习闭环；
以知识库版本、检索日志和评估体系保障 AI 可靠性；
以教师和管理看板体现企业培训价值。
```

后续真实项目落地时，应先做可验证、可追踪、可运营的 MVP，再逐步增强 AI 效果和平台能力。
