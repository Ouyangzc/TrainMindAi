# 课程资料上传与文件服务设计方案

## 1. 已确认决策

| 序号 | 决策项 | 结论 |
|---|---|---|
| 1 | 文件归属对象 | 文件归属课程下的模块，允许课程公共资料 |
| 2 | 文档版本 | 采用“文档 + 文档版本”模型 |
| 3 | 文件类型 | 第一阶段仅支持 `pdf`、`docx`、`pptx`、`xlsx` |
| 4 | 文件大小 | 单文件最大 200MB |
| 5 | 权限模型 | 管理员和课程讲师/负责人可管理资料，学员只读，不加入助教 |
| 6 | 解析触发 | 上传后不自动解析，由管理员或课程讲师/负责人手动触发 |
| 7 | 前端入口 | `课程管理 -> 课程详情 -> 资料管理 Tab` |
| 8 | 文件预览 | 第一阶段不做在线预览，只支持下载原文件 |
| 9 | 学员端状态 | 学员只看到当前已发布知识库版本纳入的资料，不展示解析状态 |
| 10 | 多租户 | 表结构预留 `tenant_id`，第一阶段默认单租户 |
| 11 | 状态字典 | 状态展示和筛选统一接入 RuoYi 字典，后端保留业务常量和流转校验 |

## 2. 业务范围

第一阶段目标是打通课程资料上传到 AI 解析的闭环：

```text
课程 -> 模块 -> 文档 -> 文档版本 -> SeaweedFS 对象 -> 手动 AI 解析任务
```

本方案只负责“资料文件服务”边界，终点是文档版本解析完成。解析完成的资料不会自动进入学员可见范围，也不会自动用于 AI 问答。解析后的资料如何组成知识快照、构建索引、发布和回滚，由《知识库治理设计方案》负责。

第一阶段不做：

- 在线预览。
- 视频、图片、压缩包、纯文本解析。
- 学员上传资料。
- 完整多租户管理。
- 自动解析。
- 断点续传。
- 知识库草稿、构建、发布和回滚。
- Chunk、Embedding 和关键词索引管理。

## 3. 领域模型

### 3.1 课程

课程是培训业务的最大组织单元，例如“航空维修培训”。

`course.owner_user_id` 是课程唯一主负责人的权威字段。创建课程时必须同步创建该用户的 `course_user(owner)` 授权；主负责人变更必须通过“转移负责人”操作在同一事务中同步两处数据。

### 3.2 模块

模块是课程下的结构单元，例如 M1 到 M8。

```text
航空维修培训
  ├─ M1
  ├─ M2
  ├─ ...
  └─ M8
```

模块用于：

- 组织课程资料。
- 作为前端筛选维度。
- 作为 AI 检索和学习统计的业务维度。

### 3.3 文档

文档是一个业务资料概念，例如“M1 上课 PPT”。

文档本身不直接等同于物理文件。文档下面可以有多个版本。

### 3.4 文档版本

文档版本代表一次上传的具体文件。例如：

```text
M1 上课 PPT
  ├─ v1
  ├─ v2
  └─ v3 当前版本
```

每个版本对应 SeaweedFS 中一个独立对象。

文档版本不是用户手动填写版本号创建的独立业务对象，而是由系统在以下操作中自动创建：

- 用户首次上传资料时，系统自动创建文档和 v1 版本。
- 管理员或课程讲师/负责人点击“上传新版本”并上传文件时，系统自动创建下一个版本号。

用户只选择“上传资料”或“上传新版本”，不需要手工输入 `version_no`。

## 4. 状态设计

### 4.1 文档状态

`course_document.status`

| 状态 | 含义 |
|---|---|
| `active` | 正常 |
| `archived` | 归档 |

归档状态表示资料不再作为当前课程资料继续维护，但仍保留历史记录和版本文件。例如某个模块资料已经被新版课程体系替换，管理员不希望它出现在默认资料列表，也不希望继续被更新，但后续审计、回溯或历史课程仍可能需要查看。归档不是删除，归档资料仍保留数据库记录和对象存储文件。

逻辑删除统一由 `del_flag` 字段表达，不依赖 `status = deleted`。因此 `status` 只表达业务状态，`del_flag` 表达是否删除。

第一阶段文档业务状态可以只使用 `active` 和 `archived`，删除统一设置 `del_flag = '2'`。

### 4.2 文档版本状态

`course_document_version.status`

| 状态 | 含义 | 可纳入知识库草稿 |
|---|---|---|
| `uploaded` | 已上传，待解析 | 否 |
| `parsing` | 解析中 | 否 |
| `parsed` | 已解析 | 是 |
| `failed` | 处理失败 | 否 |
| `archived` | 历史归档 | 否 |

说明：

- `parsed` 只表示单个文件解析成功，不表示已经发布。
- 管理员和课程讲师/负责人可看到版本状态。
- 只有未删除且状态为 `parsed` 的资料版本可以加入知识库草稿。
- 学员资料可见性和 AI 问答范围由当前 `published` 的知识库版本决定。

### 4.3 RuoYi 字典设计

状态类字段接入 RuoYi 字典，前端列表、筛选下拉、状态 Tag 和导出显示使用字典。后端业务流转不能只依赖字典文本，应保留常量或枚举类做校验，避免字典被误改后破坏解析和权限逻辑。

建议新增字典类型：

| 字典类型 | 字典名称 | 用途 |
|---|---|---|
| `trainmind_course_status` | 课程状态 | `course.status` |
| `trainmind_course_module_status` | 课程模块状态 | `course_module.status` |
| `trainmind_course_document_status` | 课程资料状态 | `course_document.status` |
| `trainmind_document_version_status` | 资料版本状态 | `course_document_version.status` |
| `trainmind_document_parse_task_status` | 文档解析任务状态 | `document_parse_task.status` |

建议字典数据：

`trainmind_course_status`

| 标签 | 值 | 样式 |
|---|---|---|
| 草稿 | `draft` | `info` |
| 正常 | `active` | `success` |
| 停用 | `disabled` | `danger` |

`trainmind_course_module_status`

| 标签 | 值 | 样式 |
|---|---|---|
| 正常 | `active` | `success` |
| 停用 | `disabled` | `danger` |

`trainmind_course_document_status`

| 标签 | 值 | 样式 |
|---|---|---|
| 正常 | `active` | `success` |
| 归档 | `archived` | `info` |

`trainmind_document_version_status`

| 标签 | 值 | 样式 |
|---|---|---|
| 待解析 | `uploaded` | `info` |
| 解析中 | `parsing` | `warning` |
| 已解析 | `parsed` | `success` |
| 失败 | `failed` | `danger` |
| 已归档 | `archived` | `info` |

前端使用方式：

```ts
const {
  trainmind_course_document_status,
  trainmind_document_version_status
} = useDict('trainmind_course_document_status', 'trainmind_document_version_status')
```

表格回显使用 `DictTag`：

```vue
<dict-tag :options="trainmind_document_version_status" :value="row.status" />
```

后端建议提供状态常量：

```java
public final class CourseDocumentStatus {
    public static final String ACTIVE = "active";
    public static final String ARCHIVED = "archived";
}

public final class DocumentVersionStatus {
    public static final String UPLOADED = "uploaded";
    public static final String PARSING = "parsing";
    public static final String PARSED = "parsed";
    public static final String FAILED = "failed";
    public static final String ARCHIVED = "archived";
}
```

这些常量用于权限判断、查询条件和状态流转；字典只负责展示、筛选和后台可维护的标签样式。

## 5. 表结构设计

以下表设计面向 PostgreSQL。字段命名采用下划线风格，和现有 RuoYi 表风格保持一致。

所有业务表统一包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `del_flag` | `char(1)` | 删除标志，`0` 存在，`2` 删除 |
| `remark` | `varchar(500)` | 备注 |

所有删除操作统一为逻辑删除，即更新 `del_flag = '2'`。查询默认带 `del_flag = '0'` 条件。对象存储文件不在删除接口里立即物理删除，由后续清理任务统一处理。

如果现有系统已经有课程表，优先复用现有表。若没有，建议新增：

`course`

| `id` | `bigserial` | 主键 |

| `create_by` | `varchar(64)` | 创建人 |

| `update_by` | `varchar(64)` | 更新人 |

## 6. SeaweedFS 接入设计

### 6.1 接入方式

业务系统通过 S3 兼容协议接入 SeaweedFS。

配置项：

| 配置项 | 开发环境默认值 |
|---|---|
| `storage.type` | `s3` |
| `storage.endpoint` | `http://127.0.0.1:8333` |
| `storage.accessKey` | `trainmind` |
| `storage.secretKey` | `trainmind-secret` |
| `storage.bucket` | `trainmind-docs` |
| `storage.pathStyleAccess` | `true` |

### 6.2 对象 key 规范

```text
tenants/{tenant_id}/courses/{course_id}/modules/{module_id}/documents/{document_id}/versions/{version_id}/{uuid}.{ext}
```

课程公共资料使用：

```text
tenants/{tenant_id}/courses/{course_id}/modules/public/documents/{document_id}/versions/{version_id}/{uuid}.{ext}
```

示例：

```text
tenants/1/courses/100/modules/10/documents/200/versions/300/018f6f2e-....pptx
```

实际实现时不要在对象 key 中放原始中文文件名。原始文件名只存数据库。

### 6.3 后端存储抽象

建议接口：

```java
public interface ObjectStorageService {
    StoredObject putObject(UploadObjectCommand command);

    InputStream getObject(String bucket, String objectName);

    ObjectMetadata headObject(String bucket, String objectName);

    void deleteObject(String bucket, String objectName);
}
```

实现类：

```text
S3ObjectStorageService
```

不要在业务层出现 `MinioService`、`SeaweedfsService` 这类厂商绑定命名。

## 7. 后端接口设计

### 7.1 课程详情

```http
GET /course/{courseId}
```

返回课程基本信息。

### 7.2 模块列表

```http
GET /course/{courseId}/modules
```

用于课程详情页左侧模块树和模块管理。

### 7.3 新增模块

```http
POST /course/{courseId}/modules
```

权限：

- 管理员。
- 课程讲师/负责人。

### 7.4 更新模块

```http
PUT /course/{courseId}/modules/{moduleId}
```

### 7.5 资料列表

```http
GET /course/{courseId}/documents?moduleId=&keyword=&fileExt=
```

管理员和课程讲师/负责人返回管理字段：

- 当前版本。
- 文件大小。
- 上传人。
- 上传时间。
- 解析状态。
- 操作权限。

管理端默认查询 `del_flag = '0'` 的资料，可按解析状态筛选。归档资料默认不显示，可通过筛选查看。

学员返回只读字段：

- 资料名称。
- 所属模块。
- 文件类型。
- 文件大小。
- 更新时间。
- 下载入口。

学员端不读取 `latest_version_id` 判断资料范围，而是读取课程当前已发布知识库版本中的资料快照。返回字段来源于 `knowledge_base_version_document` 关联的具体文档版本，不返回解析状态。

如果课程没有已发布知识库版本，学员资料列表为空。上传新版本、重新解析和构建草稿期间，当前已发布版本继续对学员服务。

### 7.6 上传新资料

```http
POST /course/{courseId}/documents/upload
Content-Type: multipart/form-data
```

参数：

| 参数 | 说明 |
|---|---|
| `moduleId` | 模块 ID，可为空 |
| `title` | 资料标题，可默认取文件名去扩展名 |
| `file` | 上传文件 |

处理流程：

1. 校验用户权限。
2. 校验文件类型，只允许 `pdf/docx/pptx/xlsx`。
3. 校验文件大小，不超过 200MB。
4. 计算 MD5。
5. 创建 `course_document`，`del_flag = '0'`。
6. 系统自动创建 `course_document_version` 初始记录，`version_no = 1`，`status = uploaded`，`del_flag = '0'`。
7. 生成对象 key。
8. 上传 SeaweedFS。
9. 更新版本记录的 `bucket`、`object_name`、`checksum_md5` 等字段。
10. 更新文档 `latest_version_id`。

上传成功后不自动解析。

### 7.7 上传新版本

```http
POST /course/{courseId}/documents/{documentId}/versions
Content-Type: multipart/form-data
```

处理流程同上传新资料，但不创建新的 `course_document`。系统自动计算当前文档最大版本号并创建新的 `course_document_version`：

```text
version_no = max(version_no) + 1
```

用户不手工输入版本号。

### 7.8 手动解析

```http
POST /course/{courseId}/documents/{documentId}/versions/{versionId}/parse
```

处理流程：

1. 校验用户权限。
2. 校验版本存在且属于当前课程。
3. 校验文件类型支持解析。
4. 将 `status` 更新为 `parsing`。
5. 调用 AI 服务解析接口。
6. 保存 `parse_task_id`。

AI 服务接口：

```http
POST /internal/v1/documents/{document_version_id}/parse
```

请求：

```json
{
  "course_id": 100,
  "document_id": 200,
  "object_name": "tenants/1/courses/100/modules/10/documents/200/versions/300/file.pptx",
  "file_ext": "pptx",
  "checksum_md5": "..."
}
```

### 7.9 查询解析任务状态

```http
GET /course/{courseId}/documents/{documentId}/versions/{versionId}/parse-task
```

仅管理员和课程讲师/负责人可见。

业务后端通过 AI 内部接口查询独立解析任务：

```http
GET /internal/v1/document-parse-tasks/{taskId}
```

### 7.10 下载原文件

```http
GET /course/{courseId}/documents/{documentId}/versions/{versionId}/download
```

第一阶段建议由后端代理下载：

1. 校验用户是否有课程访问权限。
2. 从 SeaweedFS 获取对象流。
3. 设置下载文件名为 `original_filename`。
4. 返回文件流。

### 7.11 删除资料

```http
DELETE /course/{courseId}/documents/{documentId}
```

第一阶段软删除：

- `course_document.del_flag = '2'`
- 该文档下的 `course_document_version.del_flag = '2'`
- 不立即删除 SeaweedFS 对象
- 后续由后台清理任务处理物理对象

删除前必须检查知识库版本引用：

- 删除接口先检查该资料的全部文档版本。
- 任一版本被活动、已发布或保留的历史知识库版本引用时，拒绝整个删除请求。
- 全部版本均无引用时，才统一逻辑删除文档及其所有版本。
- 物理清理对象前必须确认不存在任何保留中的知识库版本引用。

## 10. AI 解析状态同步

手动解析后，业务后端创建 `document_parse_task` 并保存 `parse_task_id`。

解析任务只绑定 `document_version`，不要求先创建 `knowledge_base_version`。解析成功后版本进入 `parsed`，等待知识库治理边界选择纳入某个知识库草稿。

状态同步可分两阶段：

第一阶段：

- 前端轮询业务后端。
- 业务后端查询 AI 服务 `document_parse_task` 状态。
- 更新 `course_document_version.status`。

第二阶段：

- AI 服务回调业务后端。
- 或业务后端定时同步任务状态。

## 11. 异常和回滚

### 11.1 上传对象失败

处理：

- 文档版本标记为失败或删除草稿记录。
- 返回明确错误。
- 不创建可见资料。

第一阶段建议在事务中先创建记录，再上传对象。若上传失败，回滚数据库事务。

### 11.2 数据库写入失败

如果对象已经上传但数据库写入失败：

- 记录错误日志。
- 后续后台清理孤儿对象。

### 11.3 AI 解析失败

处理：

- `status = failed`
- 保存 `parse_error_message`
- 管理端显示失败原因。
- 支持重新解析。
