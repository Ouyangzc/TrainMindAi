# 课程资料与知识库治理实施计划

## 1. 目标

基于《企业级知识助教系统落地方案》，按两个边界实施：

- 资料文件服务：课程资料上传、版本管理、SeaweedFS 存储、手动解析和原文件下载。
- 知识库治理：选择已解析资料版本、构建知识快照、人工发布和回滚。

详细设计分别见《课程资料上传与文件服务设计方案》《知识库治理设计方案》和《文件服务落地计划（SeaweedFS）》。

第一阶段只做确定范围：

- 课程下模块资料管理，支持课程公共资料。
- 文档 + 文档版本模型。
- 支持 `pdf`、`docx`、`pptx`、`xlsx`。
- 单文件最大 200MB。
- 使用 SeaweedFS S3 兼容接口。
- 上传后不自动解析，由管理员或课程讲师/负责人手动解析。
- 学员只读，只能看到当前已发布知识库版本纳入的资料，不展示解析状态。
- 状态展示和筛选使用 RuoYi 字典。
- 删除统一逻辑删除，使用 `del_flag = '2'`。
- `parsed` 只代表文件解析成功，不代表对学员发布。

## 2. 前置条件

实施前需要确认以下环境可用：

| 项目 | 要求 |
|---|---|
| PostgreSQL | 本机 `ruoyi` 数据库可连接 |
| SeaweedFS | S3 Endpoint `http://127.0.0.1:8333` 可访问 |
| Bucket | `trainmind-docs` 已初始化 |
| 后端环境 | JDK、Maven 可用 |
| 前端环境 | Node、包管理器可用 |
| AI 服务 | 至少具备内部解析接口或可先使用联调桩 |

SeaweedFS 本机服务当前按以下配置接入：

| 配置项 | 默认值 |
|---|---|
| Endpoint | `http://127.0.0.1:8333` |
| Bucket | `trainmind-docs` |
| Access Key | `trainmind` |
| Secret Key | `trainmind-secret` |
| Path Style | `true` |

MVP 默认由 Java 后端和 AI 服务共用上述业务 S3 凭据。Admin UI 使用独立管理员凭据。默认值只用于本地开发，测试和生产环境必须通过外部配置注入，检测到默认密钥时拒绝启动。

课程成员接口：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/course/{courseId}/members` | `GET` | 查询课程成员 |
| `/course/{courseId}/members` | `POST` | 添加直接授权 |

核心规则：

- 课程、知识库和课程创建人的 `owner` 授权必须在同一事务中成功或回滚。
- `course.owner_user_id` 是唯一主负责人权威字段，必须与唯一有效的 `course_user(owner)` 一致。
- 首次草稿从空快照开始；后续草稿自动复制当前发布版本的资料快照关联。
- 同一知识库的 `draft / building / ready / failed` 状态合计最多存在一个活动版本。
- 只有未删除且状态为 `parsed` 的资料版本可以加入草稿；发布知识库不会修改资料版本状态。
- 构建开始后冻结资料快照。
- 创建草稿时写入系统默认的 Chunk 和检索策略版本，页面只读展示。
- MVP 对新知识库版本中的全部资料重新生成 Chunk、Embedding 和关键词索引，不复用历史 Chunk。
- Java 发起构建时只传 `knowledge_base_version_id`；AI 使用共享 PostgreSQL 只读查询该版本全部有效资料快照。
- AI 必须在遍历资料前一次性清理目标版本旧 Chunk，不能在每个资料循环内重复清理。
- `ready` 版本必须人工发布。
- 发布事务更新新旧版本状态和 `knowledge_base.current_version_id`。
- 新版本构建失败时当前发布版本继续服务。
- 学员资料和 AI 问答读取同一个当前发布版本。
- 回滚不直接激活历史版本，而是复制历史资料快照创建新的递增版本，再重新构建和发布。
- 发布和回滚通过 RuoYi 角色分配 `course:knowledge-base:publish` 权限，并同时校验课程数据权限。
- 课程没有 `current_version_id` 时，学员资料返回空集合且 AI 问答禁用。
- 学员访问资料和问答前必须存在有效 `course_user` 授权，不默认开放全部课程。

验收标准：

- 创建课程后自动生成唯一知识库基本记录和创建人的有效 `owner` 授权，`current_version_id` 为空。
- 已有活动版本时不能并发创建第二个草稿。
- 上传或解析新资料不影响当前学员资料。
- 创建后续草稿时完整复制当前发布资料快照，但不复制文件、解析结果、Chunk 和索引。
- 修改一个课件后，新知识库版本仍全量重建快照内全部资料的 Chunk 和索引。
- 构建成功但未发布的版本对学员不可见。
- 发布后学员资料和问答同时切换。
- 历史版本回滚时生成新的版本号，原历史版本保持不可变。
- 已发布资料快照不会被逻辑删除破坏。
- 全量构建完成后，每个有效资料快照都存在对应 Chunk，不能只保留最后一个资料的结果。
- 无有效 `course_user` 授权时，学员访问课程资料和问答被拒绝。
- 主负责人不能通过普通成员编辑接口停用、移除或修改角色，只能通过负责人转移接口变更。

## 5. 端到端验收用例

### 5.1 管理端上传资料

步骤：

1. 管理员进入课程详情。
2. 进入资料管理。
3. 选择 M1 模块。
4. 上传 `pptx` 文件。
5. 查看资料列表。

期望：

- 数据库新增 `course_document`。
- 数据库新增 `course_document_version`，`version_no = 1`。
- 版本状态为 `uploaded`。
- SeaweedFS 中存在对象。
- 管理端显示“待解析”。
- 学员端资料范围保持不变。

### 5.2 创建课程并初始化知识库

步骤：

1. 管理员创建课程。
2. 查询课程和知识库记录。

期望：

- 课程和知识库在同一事务中创建。
- 同一事务创建课程创建人的 `course_user`，角色为 `owner`，状态为 `active`。
- `tenant_id + course_id` 对应唯一知识库。
- 知识库名称默认为“{课程名称}知识库”。
- `current_version_id` 为空。
- 不自动创建知识库版本。

### 5.3 课程成员授权

步骤：

1. 管理员在课程成员 Tab 添加学员。
2. 设置访问角色、状态和有效期。
3. 学员访问课程资料和 AI 问答。
4. 管理员停用该授权后再次访问。

期望：

- 同一课程不能重复授权同一用户。
- 有效授权且存在当前发布版本时，学员可以访问资料和问答。
- 授权停用、未生效、已失效或逻辑删除后，访问被拒绝。
- 未授权学员不能访问该课程。
- 课程主负责人只能有一个，且与 `course.owner_user_id` 一致。
- 转移负责人后，原负责人不再是 `owner`，新负责人获得有效 `owner` 授权。

### 5.4 手动解析资料

步骤：

1. 管理员点击“手动解析”。
2. 后端调用 AI 服务。
3. 等待解析完成。

期望：

- 解析中状态为 `parsing`。
- 成功后状态为 `parsed`。
- 该资料可以加入知识库草稿，但学员端仍不可见。
- 学员端不展示解析状态。

### 5.5 上传新版本

步骤：

1. 管理员对已有资料点击“上传新版本”。
2. 上传同类型或支持类型的新文件。
3. 查看版本历史。

期望：

- 新增版本 `version_no = max(version_no) + 1`。
- `latest_version_id` 指向新版本。
- 新版本状态为 `uploaded`。
- 旧版本记录保留。

### 5.6 下载资料

步骤：

1. 管理员或学员点击下载。
2. 后端代理读取 SeaweedFS 对象。

期望：

- 下载文件名使用 `original_filename`。
- 文件内容和上传文件一致。
- 学员只能下载当前已发布知识库版本纳入的资料版本。

### 5.7 发布知识库版本

步骤：

1. 管理员创建知识库草稿并选择已解析资料版本。
2. 发起构建并等待状态进入 `ready`。
3. 管理员人工确认发布。
4. 学员刷新资料列表并发起问答。

期望：

- 未发布前学员资料和问答仍使用旧 `current_version_id`。
- 发布事务更新新旧版本状态及 `knowledge_base.current_version_id`。
- 发布后学员资料列表和 AI 问答使用同一个新版本。
- 新版本构建失败时旧发布版本继续服务。

### 5.8 修改课件并全量重建

步骤：

1. 当前知识库 V1 已发布。
2. 上传并解析某个课件的新文档版本。
3. 创建 V2 草稿并确认系统复制 V1 资料快照。
4. 将草稿中的旧课件版本替换为新版本。
5. 发起构建并发布。

期望：

- V1 在 V2 构建和发布前持续服务。
- V2 对快照内全部资料重新生成 Chunk、Embedding 和关键词索引。
- V2 不复用 V1 的 Chunk 和索引记录。
- 发布后资料列表和问答同时切换到 V2。

### 5.9 基于历史版本回滚

步骤：

1. 在历史版本中选择 V1。
2. 点击“基于此版本创建回滚版本”。
3. 检查新草稿、构建并发布。

期望：

- 系统复制 V1 资料快照创建新的递增版本，不修改 V1。
- 新回滚版本重新执行全量构建。
- 人工发布前当前线上版本持续服务。

### 5.10 无发布版本的学员端

期望：

- 学员仍可进入已授权课程。
- 资料列表为空。
- AI 问答入口禁用。
- 不回退读取最新解析资料或草稿资料。

### 5.11 逻辑删除资料

步骤：

1. 管理员删除资料。
2. 查询资料列表。
3. 检查 SeaweedFS 对象。

期望：

- 删除前查询该资料全部版本的知识库快照引用。
- 任一版本仍被活动、已发布或保留的历史知识库版本引用时，整个删除请求失败且不修改任何 `del_flag`。
- `course_document.del_flag = '2'`。
- 对应版本 `del_flag = '2'`。
- 默认列表不再显示。
- SeaweedFS 对象不立即物理删除。
- 如果资料版本仍被保留中的知识库版本引用，删除请求应被拒绝。

## 8. 风险控制

| 风险 | 控制方式 |
|---|---|
| 字典被误改导致业务异常 | 后端保留状态常量和流转校验，字典只负责展示 |
| 上传对象成功但数据库失败 | 记录日志，后续清理孤儿对象 |
| 数据库成功但对象上传失败 | 事务回滚或标记失败，不展示给用户 |
| AI 解析失败 | 保存失败原因，支持重新解析 |
| SeaweedFS S3 兼容差异 | 增加 S3 兼容冒烟测试 |
| 大文件上传超时 | 第一阶段限制 200MB 并使用单次 `PutObject`，失败后整体重传；后续再做 S3 Multipart Upload |
